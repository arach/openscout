## 1. Must-Fix Plan Issues

### A. `actionId` stored in `FlightRecord.metadata` is not a durable join predicate

Track 3 says: "Store `actionId` on `FlightRecord.metadata` and invocation metadata during the transition period."

`FlightRecord` is a journal projection, not an authoritative source. If the broker writes the durable `ask` action in `acceptInvocationDurably` and then writes `FlightRecord.metadata` with the `actionId` in a second step, a restart between the two leaves an orphaned durable action with no join back to the flight.

**Fix:** The join predicate on recovery must be the idempotency key, not `actionId` in metadata. The store needs `getByIdempotencyKey(key)` before any broker code writes idempotency keys. Store `actionId` in invocation metadata (which is written atomically with the invocation record), not in `FlightRecord.metadata`.

### B. First-slice stale-owner protection has a gap

Track 3, completion criterion: "A stale execution owner cannot overwrite a newer terminal result."

But Track 3 also says: "Preserve current `recordFlightDurably` calls as the public projection writer during the first slice."

`recordFlightDurably` has no lease generation check. A background task in `activeInvocationTasks` that survives across a re-claim (e.g., slow local execution while a second claim promotes a new generation) can still call `recordFlightDurably(completed)` and win, because the compatibility writer bypasses the durable signal's first-write-wins gate.

The completion criterion and the first-slice compatibility preservation are in direct tension. The plan must choose one of:
- (a) Route all terminal writes through the durable signal in slice 1 (no compatibility shortcut for terminal transitions).
- (b) Explicitly accept that stale-owner protection only applies to the durable ledger path in slice 1, and document the residual race.

Option (b) is acceptable as a conscious deferral, but must be written down as a known gap with a specific Track 3 follow-up task, not implied by the current "during the transition period" language.

### C. `claimDeliveryDurably` integration has an unresolved "or"

Track 4: "Change `claimDeliveryDurably` to use ledger claim semantics **or** share the same lease checks."

Option A (ledger claim is the authority) and Option B (dual-write with shared checks) have opposite consistency properties. Option B creates a window where the delivery row is claimed but the durable action lease is not yet held. Two SQLite tables cannot be updated atomically across what are currently separate operations. The plan must commit to one option before implementation starts.

Recommendation: Option A. Make the durable ledger the single authority for peer delivery claims in slice 1. The existing `DeliveryIntent.status` becomes a read projection derived from durable action state, not an independent source of truth.

### D. `peer_acked` collapses into `sent` — authority handoff state is lost

Track 1 maps `peer_acked` → `sent`. But `peer_acked` in `DeliveryIntent` means the remote broker accepted ownership of the invocation — authority has transferred. After authority handoff, the local lifecycle correctly stops tracking execution state, but the lifecycle snapshot should reflect that the invocation is now peer-owned, not just "sent." Collapsing to `sent` makes peer-forwarded asks look identical to locally-dispatched ones that reached the endpoint, which makes diagnosis of stuck cross-node asks harder.

Consider a `dispatched_to_peer` state or a `peerNodeId` field in `ScoutInvocationLifecycle` to make this explicit.

### E. `expired` must be resolved before Track 1 types are written

Open Question #1 ("Should `expired` be a first-class public invocation state now?") is unresolved, but Track 1 defines protocol types and the state vocabulary table includes `expired`. If `expired` is a read-time derived state (computed from `expiresAt` at query time), it is never written durably and clients cannot receive pushed updates for expiry. If a timeout worker must write it, that worker is not in scope.

**Fix:** Decide before writing Track 1 types. Recommendation: make `expired` a read-time derived state in 0.39 (derived from `expiresAt < now()` at query time, no durable write). Add an explicit note that a background sweep worker writing `expired` durably is deferred to a future SCO.

## 2. Missing Test Cases

### T1. Split terminal write (first-write-wins signal)
Two concurrent workers both resolve and attempt to write a terminal signal for the same ask action (same `actionId`, different result summaries). First write must win; second must be rejected or be a no-op. Verify the signal's idempotency key is scoped to `actionId`, not a random signal id.

### T2. Recovery after restart mid-claim
Broker creates a durable ask action, claims it (sets lease owner + generation), then the process simulates restart before `executeLocalInvocation` begins. On restart, the recovery path finds the action in `leased` state with an expired lease. A new worker should be able to re-claim it (new generation). The old generation's terminal write should be rejected.

### T3. Stale background task writes flight terminal state
Worker A claims ask action generation 1 and starts local execution. The broker re-claims with generation 2 (simulating a timeout or manual re-dispatch). Worker A's background task completes and calls `recordFlightDurably(completed)`. Verify behavior: in slice 1 (gap acknowledged), this write wins; after Track 3 complete, this write must be rejected. Both states need explicit tests so the Track 3 fix is verifiable.

### T4. Delivery dead-lettered while invocation completed
Ask completes successfully (`state: completed`). Notification delivery exhausts all retries (`state: dead_lettered`). `getInvocationLifecycle` must return both states independently. Test that a delivery `dead_lettered` event does not retroactively flip `ScoutInvocationLifecycle.state`.

### T5. Peer authority handoff then peer disappears
Local broker sends invocation to peer, receives `peer_acked`. Peer node then goes offline. Local lifecycle snapshot must show the invocation as peer-owned (not `working`, not `failed`). The local broker should not be able to independently mark it `failed` without a timeout or re-claim signal from the peer.

### T6. Duplicate ask idempotency
Same caller submits the same ask twice with identical request id, requester, target, and conversation context (same derived idempotency key). The second submission must return the original action and lifecycle, not create a new flight and durable action. Test that the response to the second submission is indistinguishable from a re-read of the first.

### T7. Suppressed vs cancelled delivery disambiguation
A delivery is cancelled by operator policy (should map to `suppressed`). A delivery is cancelled because the caller retracted the ask (should map to terminal metadata, not `suppressed`). The projection must produce different observable states. Test both paths produce the correct `ScoutDeliveryState`.

## 3. Suggested First Implementation Slice

The plan's 7-step order should be resequenced to prevent a schema migration mid-slice:

| Step | What | Why |
|------|------|-----|
| 1 | Protocol types: `ScoutInvocationLifecycle`, `ScoutOutcomeDelivery`, state vocabularies, projection helper signatures. Pure types, no implementation. | Stable target for all later steps. |
| 2 | Store: `getByIdempotencyKey(key)` on durable action store. | Recovery path cannot be implemented without this. Write it before any broker code writes idempotency keys. |
| 3 | Ask durable action creation in `acceptInvocationDurably`. Store `actionId` in invocation metadata (atomic with the invocation write). No flight behavior change. | Establishes the ledger record and idempotency key before projection reads it. |
| 4 | Lifecycle read projection: `getInvocationLifecycle` joining invocation + flight + durable action + delivery records. Expose at `/v1/invocations/:id/lifecycle`. Add lifecycle fields to `invocations_get` additively. | Now `actionId` is present in step 3, projection has a real join key. |
| 5 | Ask claim in `dispatchAcceptedInvocation` before background task launch. No change to `executeLocalInvocation` terminal writes (gap acknowledged per Issue B above). | Proves claim/recover cycle before terminal write path changes. |
| 6 | Peer delivery with `message_delivery` actions. Single-authority lease per Issue C fix (ledger claim only, delivery row is derived). | Isolated transport, existing attempt accounting makes it the safest first ledger integration. |
| 7 | CLI/MCP surface: `invocations_wait` and `invocations_get` show lifecycle state and delivery state separately. | Visible validation of the full lifecycle read model. |

## 4. Scope to Defer

| Item | Reason |
|------|--------|
| Web/desktop/mobile surface rendering (Track 5 beyond CLI/MCP) | No architecture risk; follows after CLI proves read model. |
| Track 6 retention and backfill | Migration concern; defer until Tracks 3 and 4 are stable. When written, clearly mark backfill projections as read-only (not ledger facts). |
| Mobile push and desktop notification delivery transports | Correctly deferred in plan; peer is the right first transport. |
| `expired` background sweep worker | Read-time derivation from `expiresAt` is sufficient for 0.39. Make this explicit in the spec to prevent implementers from reaching for a timer. |

## 5. Additional Blockers (from second review pass)

### F. `ScoutTerminalResult` is undefined — transcript ownership risk

`ScoutInvocationLifecycle` references `terminal?: ScoutTerminalResult` but neither spec doc defines what fields `ScoutTerminalResult` contains. This is the highest transcript-ownership risk in the plan: without an explicit schema, an implementer will naturally reach for the harness response text as `terminal.output` or `terminal.summary`. Full harness output stored in a Scout-owned lifecycle record is exactly the transcript ownership violation the spec forbids (Principle 8).

**Fix before Track 1 types are written:** Define `ScoutTerminalResult` explicitly with only compact, broker-owned fields — error class, exit code, a short broker-generated summary (≤ 256 chars), and timestamps. No raw harness stdout, no full task output, no conversation transcript excerpt. If callers need the full output they read it from the harness-owned transcript via `invocationId`, not from the lifecycle record.

### G. `invocations_wait` wait-condition semantics are ambiguous after Track 2

Track 2 says "Wire MCP `invocations_wait` … to prefer the lifecycle projection while preserving existing `flight` fields for compatibility."

Currently `invocations_wait` unblocks on flight state change (specifically `completed`, `failed`, or `cancelled`). If the implementation changes the wait condition to unblock on lifecycle state (which is a superset — it includes delivery state), calls that were previously bounded to flight completion could now wait longer (e.g., until delivery reaches `sent`). Conversely, if the wait condition only watches `ScoutInvocationState` (execution) but the response now also includes `ScoutDeliveryState`, callers that poll `invocations_wait` and inspect the response for terminal delivery state could get stale delivery state from the first return.

**Fix:** Specify explicitly that `invocations_wait` unblocks on invocation execution state changes only (`completed`, `failed`, `cancelled`, `expired`). Delivery state is included in the response snapshot but does not extend the wait. Document this in the Track 2 scope so the MCP implementation is unambiguous.

---

## Summary

The plan is structurally sound and the principle of building on SCO-027's ledger rather than inventing a new workflow runtime is the right call. The main risk is that the "transition period" language in Track 3 leaves a real stale-owner correctness gap in the first slice without naming it explicitly. The `claimDeliveryDurably` "or" in Track 4 must be resolved to Option A before implementation starts. The `peer_acked` → `sent` mapping should be revisited before the protocol types are finalized. Everything else is missing test coverage or deferred scope that is safe to call out now.

### One note on journal vs store lookup

`broker-journal.ts getDurableActionByIdempotencyKey` iterates all durable actions linearly. This is fine for journal replay (bounded, transient structure), but if the journal is ever used as a read path during steady-state operation, this becomes O(n). The SQLite store path is the right primary lookup; the journal implementation should stay as-is but not be promoted to a steady-state read path.
