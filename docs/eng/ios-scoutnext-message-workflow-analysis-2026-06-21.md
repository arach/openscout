# iOS ScoutNext message workflow analysis

Date: 2026-06-21  
Scope: the current end-to-end message path for the iOS ScoutNext conversation surface: compose/send, bridge RPC transport, live delivery, projection, and markdown rendering.

## Executive summary

The iOS conversation path is built around a clean split: `ConversationSurface` owns SwiftUI state and UX, `BridgeBrokerClient` maps shared capability calls onto bridge RPC methods, `BridgeConnection` owns encrypted WebSocket transport and event fanout, `ConversationProjection` is the pure reducer, and `MessageMarkupParser` / `MessageMarkupView` parse and render body text natively (`apps/ios/Scout/ConversationSurface.swift:9-13`, `packages/scout-ios-core/Sources/ScoutIOSCore/BridgeBrokerClient.swift:1-15`, `packages/scout-ios-core/Sources/ScoutIOSCore/BridgeConnection.swift:1-21`, `packages/scout-native-core/Sources/ScoutCapabilities/ConversationProjection.swift:1-18`, `packages/scout-native-core/Sources/ScoutCapabilities/MessageMarkup.swift:1-9`, `apps/ios/Scout/MessageMarkupView.swift:8-13`).

The current path works as a snapshot-first, then live-event stream for session events, but broker comms sends are currently reconciled by polling snapshots because the surface itself notes that “Comms conversations get no live session-event push” (`apps/ios/Scout/ConversationSurface.swift:40-42`, `apps/ios/Scout/ConversationSurface.swift:525-527`, `apps/ios/Scout/ConversationSurface.swift:567-583`). This split is the main architectural seam to fix: session lifecycle events are push-based, while broker conversation messages are snapshot/poll-based.

The highest-impact risks are: the streaming badge is set true on any event and only false when the stream ends rather than after the reducer clears `currentTurnId`; iOS drops connection state without auto-reconnect/replay; retry can create another long-lived stream consumer; optimistic send reconciliation uses message text/signature rather than the `clientMessageId`; and the parser only detects triple-backtick code fences, not tilde fences, indented code, or unfenced JSON/object payloads (`apps/ios/Scout/ConversationSurface.swift:468-497`, `packages/scout-native-core/Sources/ScoutCapabilities/ConversationProjection.swift:115-138`, `packages/scout-ios-core/Sources/ScoutIOSCore/BridgeConnection.swift:16-18`, `packages/scout-ios-core/Sources/ScoutIOSCore/BridgeConnection.swift:961-1002`, `apps/ios/Scout/ConversationSurface.swift:388-390`, `apps/ios/Scout/ConversationSurface.swift:589-607`, `packages/scout-native-core/Sources/ScoutCapabilities/MessageMarkup.swift:155-160`, `packages/scout-native-core/Sources/ScoutCapabilities/MessageMarkup.swift:218-230`).

## Current end-to-end workflow

### 1. Compose and dispatch on iOS

`ConversationSurface` owns the input state: `composerText`, `isSending`, selected attachments, file importer state, `composerError`, optimistic `pending` sends, and the `replyPoll` task (`apps/ios/Scout/ConversationSurface.swift:26-42`). The composer is installed as a bottom safe-area inset on the conversation view, and the surface starts `run()` with `.task(id: conversationId)` (`apps/ios/Scout/ConversationSurface.swift:67-75`).

The composer enables send only when there is trimmed text or at least one attachment and no send is already in progress (`apps/ios/Scout/ConversationSurface.swift:172-174`). Attachments are gathered from `PhotosPicker` and the file importer into `ScoutComposerAttachment` values, with errors surfaced in `composerError` (`apps/ios/Scout/ConversationSurface.swift:163-168`, `apps/ios/Scout/ConversationSurface.swift:271-300`, `apps/ios/Scout/MessageMarkupView.swift:202-215`). Dictation appends final utterances into the same composer text field (`apps/ios/Scout/ConversationSurface.swift:227-231`, `apps/ios/Scout/ConversationSurface.swift:263-269`).

On `send()`, iOS trims text, captures attachments, clears the composer, resets `composerError`, marks `isSending`, cancels the old poll task, uploads attachments, inserts an optimistic local user turn, calls `client.send(PromptSpec(...))`, then polls snapshots until a non-user agent reply has landed or a 120-second deadline expires (`apps/ios/Scout/ConversationSurface.swift:500-535`, `apps/ios/Scout/ConversationSurface.swift:537-565`, `apps/ios/Scout/ConversationSurface.swift:567-583`). Send errors now restore the text/attachments and surface the localized error in the composer (`apps/ios/Scout/ConversationSurface.swift:528-533`).

The client-side send path is `BridgeBrokerClient.send(_:)`, which converts the prompt into `MobileCommsSendParams`, generates a fresh `clientMessageId`, and calls the donor method name `mobile/comms/send` through `BridgeConnection.rpc` (`packages/scout-ios-core/Sources/ScoutIOSCore/BridgeBrokerClient.swift:184-188`, `packages/scout-ios-core/Sources/ScoutIOSCore/BridgeBrokerClient.swift:310-318`). `RPCWire` maps that donor method to the tRPC path `mobile.commsSend` as a mutation and encodes requests as JSON-RPC 2.0 tRPC envelopes (`packages/scout-ios-core/Sources/ScoutIOSCore/RPCWire.swift:35-52`, `packages/scout-ios-core/Sources/ScoutIOSCore/RPCWire.swift:54-81`).

On the bridge side, `mobile.commsSend` is routed by the tRPC router to `sendScoutMobileComms(input, resolveMobileCurrentDirectory(), ctx.deviceId)` (`packages/web/server/core/pairing/runtime/bridge/router.ts:1042-1064`). `sendScoutMobileComms` validates broker context and conversation existence, routes 1:1 direct messages through `sendScoutMobileMessage`, routes group/thread posts through `sendScoutConversationMessage`, and routes channels through `sendScoutMessage` (`packages/web/server/core/mobile/service.ts:1298-1333`, `packages/web/server/core/mobile/service.ts:1336-1365`).

The snapshot read model converts broker messages back into `SessionState` turns: `getScoutMobileSessionSnapshot` resolves a conversation id or bare agent id, pages recent messages, maps each broker message to a completed text or reasoning block plus file blocks, and appends a synthetic streaming “Working…” turn when an active flight is newer than the last agent message (`packages/web/server/core/mobile/service.ts:655-689`, `packages/web/server/core/mobile/service.ts:691-712`, `packages/web/server/core/mobile/service.ts:727-756`, `packages/web/server/core/mobile/service.ts:758-812`).

### 2. Bridge transport and delivery

The desktop pairing runtime starts a `Bridge`, the tRPC bridge server, the attachment file server, and an outbound relay connection (`packages/web/server/core/pairing/runtime/runtime.ts:62-96`). The `Bridge` wraps the session registry, exposes send/interrupt/answer/decide, returns snapshots, emits sequenced events, and can replay buffered events (`packages/web/server/core/pairing/runtime/bridge/bridge.ts:42-65`, `packages/web/server/core/pairing/runtime/bridge/bridge.ts:72-93`, `packages/web/server/core/pairing/runtime/bridge/bridge.ts:103-139`).

The relay is intentionally an opaque WebSocket room forwarder: it tracks one bridge socket, multiple client sockets, a bridge public-key to room map for `/resolve`, and forwards bridge/client payloads without interpreting Pairing or session semantics (`packages/web/server/core/pairing/runtime/relay/relay.ts:1-9`, `packages/web/server/core/pairing/runtime/relay/relay.ts:49-74`, `packages/web/server/core/pairing/runtime/relay/relay.ts:83-104`, `packages/web/server/core/pairing/runtime/relay/relay.ts:140-170`). When the bridge leaves, clients are closed after a grace period and the room/key index is cleaned up (`packages/web/server/core/pairing/runtime/relay/relay.ts:178-195`).

The bridge connects outbound to the relay as the `bridge` role, exposes a QR payload for the phone, auto-reconnects to the relay with exponential backoff, and maintains one Noise transport per mobile client so multiple phones or reconnects cannot share cipher state (`packages/web/server/core/pairing/runtime/bridge/relay-client.ts:1-10`, `packages/web/server/core/pairing/runtime/bridge/relay-client.ts:86-120`, `packages/web/server/core/pairing/runtime/bridge/relay-client.ts:190-226`).

On iOS, `BridgeConnection.connect()` loads the phone identity and trusted bridge info, recovers Bonjour-discovered relay routes when needed, validates the trusted bridge key, assembles relay candidates, resolves the bridge’s current room through `/resolve`, performs a Noise IK handshake to the known bridge key, persists the winning route, and starts the receive loop (`packages/scout-ios-core/Sources/ScoutIOSCore/BridgeConnection.swift:403-492`, `packages/scout-ios-core/Sources/ScoutIOSCore/BridgeConnection.swift:687-815`, `packages/scout-ios-core/Sources/ScoutIOSCore/BridgeConnection.swift:838-880`). First-time pairing uses the QR payload, Bonjour plus QR relay candidates, and Noise XX to learn and persist bridge trust (`packages/scout-ios-core/Sources/ScoutIOSCore/BridgeConnection.swift:494-590`).

`SecureTransport` defines the encrypted frame shape as `{phase:"handshake"|"transport", payload:<base64>}`, makes the phone the Noise initiator, chooses IK when a remote static key is known and XX for pairing, and derives a transport session after a 3-second bounded handshake (`packages/scout-ios-core/Sources/ScoutIOSCore/SecureTransport.swift:1-12`, `packages/scout-ios-core/Sources/ScoutIOSCore/SecureTransport.swift:100-151`). It serializes application sends to preserve cipher nonce/write order and decrypts incoming transport messages into an `AsyncStream<String>` (`packages/scout-ios-core/Sources/ScoutIOSCore/SecureTransport.swift:210-246`, `packages/scout-ios-core/Sources/ScoutIOSCore/SecureTransport.swift:248-312`).

The bridge server handles secure WebSockets by completing Noise as responder, rejecting untrusted direct peers, building tRPC context, subscribing the socket to bridge events, and then parsing decrypted messages as tRPC JSON-RPC (`packages/web/server/core/pairing/runtime/bridge/server-trpc.ts:589-647`). Its tRPC dispatcher handles queries/mutations as single responses and subscriptions as started/data/stopped streams (`packages/web/server/core/pairing/runtime/bridge/server-trpc.ts:234-331`, `packages/web/server/core/pairing/runtime/bridge/server-trpc.ts:335-438`). The relay-client path dispatches tRPC requests from secure peers with the peer device context (`packages/web/server/core/pairing/runtime/bridge/relay-client.ts:336-379`, `packages/web/server/core/pairing/runtime/bridge/relay-client.ts:402-427`).

Inbound session events are pushed outside the iOS explicit `conversationEvents` RPC: on connect the bridge sends all current sessions as `seq:0` `session:update` events, then forwards future `bridge.onEvent` sequenced events (`packages/web/server/core/pairing/runtime/bridge/server-trpc.ts:510-551`, `packages/web/server/core/pairing/runtime/bridge/relay-client.ts:294-334`). The server logs the expected event lifecycle — `session:update`, `turn:start`, `turn:end`, `block:start`, `block:delta`, and `block:end` — in the tRPC server event logger (`packages/web/server/core/pairing/runtime/bridge/server-trpc.ts:724-752`).

The iOS receive loop decrypts messages, routes `PING`/`PONG`, attempts to decode tRPC responses, then sequenced conversation events, then tail events, and fans decoded events to `BridgeBrokerClient` consumers (`packages/scout-ios-core/Sources/ScoutIOSCore/BridgeConnection.swift:943-959`, `packages/scout-ios-core/Sources/ScoutIOSCore/BridgeConnection.swift:1004-1031`, `packages/scout-ios-core/Sources/ScoutIOSCore/BridgeConnection.swift:1033-1049`, `packages/scout-ios-core/Sources/ScoutIOSCore/BridgeConnection.swift:1073-1105`). `BridgeBrokerClient.conversationEvents` filters those events by conversation id and `sinceSeq` before yielding them to the surface (`packages/scout-ios-core/Sources/ScoutIOSCore/BridgeBrokerClient.swift:167-179`, `packages/scout-ios-core/Sources/ScoutIOSCore/BridgeBrokerClient.swift:322-341`).

`ConversationSurface.run()` performs the UI-side contract: load an authoritative snapshot, replace the projection with that snapshot, set `loadPhase`, then fold live `conversationEvents` into `ConversationProjection` (`apps/ios/Scout/ConversationSurface.swift:468-497`). `ConversationProjection` applies snapshots wholesale, ignores `seq == 0` for the replay cursor, advances `lastAppliedSeq` for buffered events, and updates turn/block state for lifecycle events (`packages/scout-native-core/Sources/ScoutCapabilities/ConversationProjection.swift:27-39`, `packages/scout-native-core/Sources/ScoutCapabilities/ConversationProjection.swift:46-77`, `packages/scout-native-core/Sources/ScoutCapabilities/ConversationProjection.swift:100-160`, `packages/scout-native-core/Sources/ScoutCapabilities/ConversationProjection.swift:206-213`).

### 3. Body parsing and rendering

The shared conversation model defines `SessionState`, ordered `TurnState` values, and `Block` values with text, reasoning, action, file, error, and question types (`packages/scout-native-core/Sources/ScoutCapabilities/ConversationModel.swift:18-35`, `packages/scout-native-core/Sources/ScoutCapabilities/ConversationModel.swift:40-61`, `packages/scout-native-core/Sources/ScoutCapabilities/ConversationModel.swift:70-121`, `packages/scout-native-core/Sources/ScoutCapabilities/ConversationModel.swift:255-267`). `BlockView` renders text blocks with `MessageMarkupView`, reasoning as muted italic text, action/question/error/file blocks through their specialized cards, and plain error/file messages with selectable text (`apps/ios/Scout/ConversationSurface.swift:711-727`, `apps/ios/Scout/ConversationSurface.swift:742-760`, `apps/ios/Scout/ConversationSurface.swift:763-780`).

`MessageMarkupParser` turns raw markdown into semantic blocks: paragraphs, headings, rules, ordered/unordered lists, blockquotes, code, and tables (`packages/scout-native-core/Sources/ScoutCapabilities/MessageMarkup.swift:13-22`, `packages/scout-native-core/Sources/ScoutCapabilities/MessageMarkup.swift:35-52`). The current parser recognizes triple-backtick fences, horizontal rules, `#` headings, pipe tables, simple contiguous list items, blockquotes, and paragraphs (`packages/scout-native-core/Sources/ScoutCapabilities/MessageMarkup.swift:62-139`, `packages/scout-native-core/Sources/ScoutCapabilities/MessageMarkup.swift:162-215`).

`MessageMarkupView` reparses the text in its SwiftUI `body`, switches over semantic block kinds, and renders native paragraphs, headings, rules, lists, blockquotes, code blocks, and tables (`apps/ios/Scout/MessageMarkupView.swift:17-48`). Inline markdown is parsed with `.inlineOnlyPreservingWhitespace`, and inline code spans now receive a monospaced font plus a faint chip background (`apps/ios/Scout/MessageMarkupView.swift:52-75`). Fenced code blocks show the language label, preserve blank lines by rendering empty lines as a space, and keep long lines on a horizontal scroll with fixed horizontal sizing (`apps/ios/Scout/MessageMarkupView.swift:133-164`).

## Strengths

1. **Clear layering.** The UI, RPC mapping, transport, reducer, and renderer have narrow responsibilities instead of one surface owning the whole workflow (`apps/ios/Scout/ConversationSurface.swift:9-13`, `packages/scout-ios-core/Sources/ScoutIOSCore/BridgeBrokerClient.swift:1-15`, `packages/scout-ios-core/Sources/ScoutIOSCore/BridgeConnection.swift:1-21`, `packages/scout-native-core/Sources/ScoutCapabilities/ConversationProjection.swift:1-18`).

2. **Snapshot-first recovery with reducer reuse.** `run()` loads `client.snapshot`, the bridge snapshot read model reconstructs completed turns from broker messages, and `ConversationProjection` keeps a deterministic event-to-state reducer with sequence tracking (`apps/ios/Scout/ConversationSurface.swift:468-481`, `packages/web/server/core/mobile/service.ts:727-812`, `packages/scout-native-core/Sources/ScoutCapabilities/ConversationProjection.swift:27-39`).

3. **End-to-end encrypted bridge payloads.** The relay forwards opaque payloads, `BridgeConnection` verifies the trusted bridge key, and `SecureTransport` wraps tRPC/event frames inside Noise transport frames (`packages/web/server/core/pairing/runtime/relay/relay.ts:1-9`, `packages/scout-ios-core/Sources/ScoutIOSCore/BridgeConnection.swift:443-465`, `packages/scout-ios-core/Sources/ScoutIOSCore/SecureTransport.swift:1-12`, `packages/scout-ios-core/Sources/ScoutIOSCore/SecureTransport.swift:210-246`).

4. **Optimistic UX with an actual send failure path.** The surface inserts a local user turn after attachment upload succeeds, then restores composer input and shows `composerError` on send failure (`apps/ios/Scout/ConversationSurface.swift:500-533`, `apps/ios/Scout/ConversationSurface.swift:546-565`).

5. **Renderer improvements landed in the right layer.** Inline code styling belongs in `MessageMarkupView.inline`, while code-block blank-line preservation and horizontal line scrolling belong in `codeBlock`, so the transport and parser stay unchanged for visual-only improvements (`apps/ios/Scout/MessageMarkupView.swift:52-75`, `apps/ios/Scout/MessageMarkupView.swift:133-164`).

## Gaps and risks

### P0/P1 lifecycle and delivery risks

1. **Streaming state is not derived from the reducer after live events.** `run()` sets `isStreaming = true` for every yielded event and only sets it false after the stream terminates, but the reducer already clears `currentTurnId` on `turn:end` and `turn:error`; the surface should derive `isStreaming` from `projection.state?.currentTurnId` after applying each event (`apps/ios/Scout/ConversationSurface.swift:487-497`, `packages/scout-native-core/Sources/ScoutCapabilities/ConversationProjection.swift:115-138`). `applyRefreshed` already uses the correct snapshot-derived logic (`apps/ios/Scout/ConversationSurface.swift:622-628`).

2. **Broker comms messages are polled, not pushed, in this surface.** The code explicitly states that comms conversations get no live session-event push, so after sending it polls snapshots until the reply appears; this makes reply visibility depend on a 1.2-second polling cadence and a 120-second deadline instead of the same event stream that drives session turns (`apps/ios/Scout/ConversationSurface.swift:40-42`, `apps/ios/Scout/ConversationSurface.swift:525-527`, `apps/ios/Scout/ConversationSurface.swift:567-583`). The server can list conversations/messages and send comms, but those APIs are query/mutation routes rather than an iOS-consumed live conversation update stream (`packages/web/server/core/pairing/runtime/bridge/router.ts:1016-1064`, `packages/web/server/core/mobile/service.ts:1200-1295`).

3. **iOS has teardown on unexpected disconnect, but no reconnect/replay loop in the transport class.** The file-level comment says automatic reconnect-on-drop was dropped, `markDisconnected` clears connection state and pending requests, and the server exposes `sync.replay` / `sync.status` that could support replay; iOS’s current route map does not include those sync procedures (`packages/scout-ios-core/Sources/ScoutIOSCore/BridgeConnection.swift:16-18`, `packages/scout-ios-core/Sources/ScoutIOSCore/BridgeConnection.swift:961-1002`, `packages/web/server/core/pairing/runtime/bridge/router.ts:1422-1453`, `packages/scout-ios-core/Sources/ScoutIOSCore/RPCWire.swift:35-52`).

4. **Retry can create a second stream consumer.** The view already starts `run()` from `.task(id: conversationId)`, while the failed empty state calls `Task { await run() }` without cancelling or replacing the existing task; since `run()` contains a long-lived `for await` event loop, a manual retry can produce duplicate consumers for the same conversation (`apps/ios/Scout/ConversationSurface.swift:75`, `apps/ios/Scout/ConversationSurface.swift:374-390`, `apps/ios/Scout/ConversationSurface.swift:468-497`).

5. **Optimistic reconciliation can collide on repeated text.** The local optimistic id is generated in `PendingUserSend`, but `applyRefreshed` removes pending sends when the snapshot contains the same user text or text/file signature; the bridge send path separately creates a `clientMessageId` that is not carried back into the pending reconciliation key (`apps/ios/Scout/ConversationSurface.swift:54-65`, `apps/ios/Scout/ConversationSurface.swift:517-521`, `apps/ios/Scout/ConversationSurface.swift:589-607`, `packages/scout-ios-core/Sources/ScoutIOSCore/BridgeBrokerClient.swift:184-188`, `packages/scout-ios-core/Sources/ScoutIOSCore/BridgeBrokerClient.swift:310-318`).

6. **Some control errors are swallowed.** `send()` now restores state and shows `composerError`, but `answer()` and `decide()` still call their RPCs with `try?` and no user-visible failure state (`apps/ios/Scout/ConversationSurface.swift:528-533`, `apps/ios/Scout/ConversationSurface.swift:649-663`).

### Rendering, parser, and performance risks

7. **The parser only detects triple-backtick code fences.** `fenceLanguage` returns a language only when the trimmed line starts with ```; `isBlockStart` delegates to that same helper, so tilde fences and indented code blocks are currently parsed as paragraphs or other simpler blocks (`packages/scout-native-core/Sources/ScoutCapabilities/MessageMarkup.swift:155-160`, `packages/scout-native-core/Sources/ScoutCapabilities/MessageMarkup.swift:218-230`).

8. **Unfenced JSON/object bodies are not promoted to code or structured blocks.** The parser’s public block kinds do not include JSON/object, and the parse decision tree falls through from fences/rules/headings/tables/lists/quotes into a paragraph fallback (`packages/scout-native-core/Sources/ScoutCapabilities/MessageMarkup.swift:13-22`, `packages/scout-native-core/Sources/ScoutCapabilities/MessageMarkup.swift:62-139`).

9. **Long-thread support exists server-side but has no visible “load older” affordance here.** The mobile service pages recent snapshots at 24 messages and history pages at 40 messages, and returns `history.hasOlder` / `oldestTurnId`; the iOS transcript renders `turns` in a `LazyVStack` and auto-scrolls on every last-block text/count/turn-count change, but this file has no call that requests `beforeTurnId` from the surface (`packages/web/server/core/mobile/service.ts:179-180`, `packages/web/server/core/mobile/service.ts:474-520`, `packages/web/server/core/mobile/service.ts:805-812`, `apps/ios/Scout/ConversationSurface.swift:404-424`, `packages/scout-ios-core/Sources/ScoutIOSCore/BridgeBrokerClient.swift:161-165`).

10. **Streaming deltas can reparse full text in SwiftUI body.** The reducer appends each `block:delta` to the block text, `ConversationSurface` auto-scrolls on last block text changes, and `MessageMarkupView.body` reparses the full string every time SwiftUI reevaluates the view (`packages/scout-native-core/Sources/ScoutCapabilities/ConversationProjection.swift:151-160`, `apps/ios/Scout/ConversationSurface.swift:421-424`, `apps/ios/Scout/MessageMarkupView.swift:17-22`).

11. **Text is selectable, but there are no explicit message-level copy affordances in these components.** Paragraphs, headings, list items, blockquotes, code lines, and error cards use `.textSelection(.enabled)`, but the rendering code shown here provides no message-level context menu, code-block copy button, or share action (`apps/ios/Scout/ConversationSurface.swift:744-752`, `apps/ios/Scout/MessageMarkupView.swift:77-85`, `apps/ios/Scout/MessageMarkupView.swift:87-97`, `apps/ios/Scout/MessageMarkupView.swift:99-117`, `apps/ios/Scout/MessageMarkupView.swift:119-130`, `apps/ios/Scout/MessageMarkupView.swift:141-149`).

12. **Tables can become cramped before code blocks do.** Code blocks get an explicit horizontal scroll, but table rows are rendered as an `HStack` of flexible cells inside the card without an equivalent horizontal scroll wrapper (`apps/ios/Scout/MessageMarkupView.swift:141-149`, `apps/ios/Scout/MessageMarkupView.swift:166-197`).

13. **Self-signed LAN TLS is accepted globally for this bridge URLSession.** `BridgeConnection` constructs a `URLSession` with `TrustAllDelegate`, and that delegate uses any server trust credential when present; that matches high-trust local pilot posture, but it should stay documented as a local bridge transport assumption rather than enterprise hardening (`packages/scout-ios-core/Sources/ScoutIOSCore/BridgeConnection.swift:390-397`, `packages/scout-ios-core/Sources/ScoutIOSCore/BridgeConnection.swift:1398-1410`).

## Ranked recommendations

1. **Fix streaming lifecycle at the source of truth.** After every snapshot or event apply, set `isStreaming = projection.state?.currentTurnId != nil` and publish the status context from that derived value; do not set streaming true merely because any event arrived (`apps/ios/Scout/ConversationSurface.swift:487-497`, `packages/scout-native-core/Sources/ScoutCapabilities/ConversationProjection.swift:115-138`). This is small, high-impact, and aligns `run()` with the already-correct `applyRefreshed` behavior (`apps/ios/Scout/ConversationSurface.swift:622-628`).

2. **Unify broker conversation delivery with live updates instead of polling.** Add a broker conversation/message event stream or snapshot invalidation stream that the iOS conversation surface can consume for comms threads, then remove the 1.2-second/120-second polling loop from the send path (`apps/ios/Scout/ConversationSurface.swift:525-527`, `apps/ios/Scout/ConversationSurface.swift:567-583`, `packages/web/server/core/pairing/runtime/bridge/router.ts:1016-1064`). The root cause is the current split between session events pushed through `bridge.onEvent` and broker comms state exposed as query/mutation APIs (`packages/web/server/core/pairing/runtime/bridge/server-trpc.ts:510-551`, `packages/web/server/core/mobile/service.ts:1200-1295`).

3. **Add iOS reconnect plus replay.** On `markDisconnected`, surface connection state and schedule a reconnect that reloads a snapshot and replays events after `projection.lastAppliedSeq` where available; wire `sync.status` / `sync.replay` into `RPCWire` or replace them with a first-class conversation subscription (`packages/scout-ios-core/Sources/ScoutIOSCore/BridgeConnection.swift:961-1002`, `packages/web/server/core/pairing/runtime/bridge/router.ts:1422-1453`, `packages/scout-ios-core/Sources/ScoutIOSCore/RPCWire.swift:35-52`). This should be implemented as a root-cause transport lifecycle fix, not as UI-only retry buttons (`packages/scout-ios-core/Sources/ScoutIOSCore/BridgeConnection.swift:16-18`).

4. **Own the `run()` task lifecycle explicitly.** Store the active stream task, cancel it before retry or conversation-id change, and split one-shot snapshot loading from long-lived event consumption so the empty-state Retry cannot spawn parallel consumers (`apps/ios/Scout/ConversationSurface.swift:75`, `apps/ios/Scout/ConversationSurface.swift:388-390`, `apps/ios/Scout/ConversationSurface.swift:468-497`).

5. **Make sends idempotent and reconcile by id.** Generate the `clientMessageId` in `ConversationSurface`, use it as the optimistic id, pass it through `PromptSpec`/`BridgeBrokerClient`, persist or echo it in broker message metadata, and remove pending sends by id instead of text/signature (`apps/ios/Scout/ConversationSurface.swift:54-65`, `apps/ios/Scout/ConversationSurface.swift:517-521`, `apps/ios/Scout/ConversationSurface.swift:589-607`, `packages/scout-ios-core/Sources/ScoutIOSCore/BridgeBrokerClient.swift:184-188`, `packages/scout-ios-core/Sources/ScoutIOSCore/BridgeBrokerClient.swift:310-318`).

6. **Expand parser coverage with fixtures before adding heuristics.** Add shared parser fixtures for triple backtick, tilde fence, indented code, unfenced JSON/object, nested/multiline lists, and tables, then extend `fenceLanguage`, `isBlockStart`, and the parse decision tree (`packages/scout-native-core/Sources/ScoutCapabilities/MessageMarkup.swift:35-52`, `packages/scout-native-core/Sources/ScoutCapabilities/MessageMarkup.swift:62-139`, `packages/scout-native-core/Sources/ScoutCapabilities/MessageMarkup.swift:155-160`, `packages/scout-native-core/Sources/ScoutCapabilities/MessageMarkup.swift:218-230`). Treat unfenced JSON as a display heuristic only when the whole block is valid JSON so ordinary prose with braces does not become code (`packages/scout-native-core/Sources/ScoutCapabilities/MessageMarkup.swift:134-139`).

7. **Reduce long-thread and streaming render cost.** Cache parsed `MessageMarkupBlock` arrays by block id plus text hash, throttle auto-scroll while deltas are arriving, and add a visible “load older” control that calls `mobile/session/snapshot` with `beforeTurnId` and merges pages into `ConversationProjection` (`apps/ios/Scout/MessageMarkupView.swift:17-22`, `apps/ios/Scout/ConversationSurface.swift:421-424`, `packages/web/server/core/mobile/service.ts:474-520`, `packages/web/server/core/mobile/service.ts:805-812`, `packages/scout-ios-core/Sources/ScoutIOSCore/BridgeBrokerClient.swift:161-165`).
