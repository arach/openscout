Baseline

As shipped today

Reference frame. Masthead (~58pt: wordmark, host plates, circled gear, hairline), full-height dock (~62pt: 16.5 glyphs, 9pt labels, lit lip), settled status strip.

What the masthead actually carries today: identity (wordmark), the host filter, and one button. None of those is inherently app-level — the filter only matters on fleet surfaces, settings is one tap anywhere, and identity is a nice-to-have on a single-purpose app you just opened on purpose.

—~120pt of every screen is chrome that mostly repeats what the surface could say itself.

Option A · Recommended

No masthead — surfaces own their context

The top bar is deleted. Each surface's first content row becomes its own contextual header: section eyebrow on the left, host filter + bare gear on the right — but only on surfaces where a host filter means something (Home, Agents, Tail). Comms or New can carry their own context (a reply target, a composer destination) in the same slot instead.

  • ~58pt of vertical space back on every surface; content starts at the safe area.
  • The filter sits next to the data it filters — cause and effect on one screen region, no app-level bar pretending all surfaces want the same context.
  • No wordmark. Identity lives in the tab glyphs, the accent, the canvas — the things that are already there.

Cost: the contextual header is per-surface work, not one shared bar — each surface opts in and the layout has to stay disciplined so the row reads as chrome, not content. Settings gear repeats per surface (or moves to a long-press on the Home tab — worth prototyping).

+Biggest reclaim; context becomes honest (each surface says what it needs).
−Surfaces must each implement the row; inconsistency is the failure mode.

Option B

Slim dock

Masthead untouched (control variable) — only the dock diet: glyphs 16.5→14, labels 9→7.5pt, padding tightened. ~62pt → ~40pt with zero structural change; selection still carried purely by the accent glyph + label.

  • Pure metrics change — `tabButtonHeight`, `tabGlyphSize`, `tabLabelSize`, `tabBarTopPadding` in ResponsiveFrame.swift. No view restructuring.
  • Labels get closer to the legibility floor, but at 7.5pt mono medium they're still above the status strip's 9pt-effective micro type… barely. The mini keeps its downscaled copy.

Trade-off: 44pt hit targets are gone — slim tabs land around 34–36pt. Acceptable for a dock you tap deliberately, worth a device pass to confirm. Ships independently of every other option.

+Cheapest win on the table — one file, four constants.
−Doesn't fix the conceptual weight, just the pixels.

Option C

Collapsible dock (shown collapsed)

Swipe down on the dock and the tab row slides away, leaving a 12pt grab strip (lit lip + pill) above the always-present status line. Swipe up from the strip — or tap it — and the tabs spring back. State persists per session; Terminal probably pins itself collapsed, Home probably never collapses.

  • The status strip is the anchor — it never moves, so the collapsed state still reads as a deliberate dock, not missing UI. The grab pill borrows the home-indicator visual language, so the affordance is familiar.
  • Collapsed = reading mode. Terminal, Tail, and long Comms threads are where ~50pt matters most — and where you're least likely to switch tabs mid-flow.

Risks: gesture collisions — surfaces scroll vertically, so the collapse gesture must start on the dock itself, not the content (same rule Safari uses). Discoverability needs one coaching moment. And a hidden tab bar hides badge/activity signals if we ever add them to tabs.

+Space on demand without committing to a smaller dock forever.
−Real interaction work: gesture, spring physics, per-surface memory.

Option D

Full reduction: A + B

No masthead, contextual surface header (~26pt), slim dock (~40pt), settled status strip. Chrome drops from ~150pt to ~66pt of fixed UI — on a 393×852 phone that's nearly a fifth of the screen handed back to content, with collapse (C) still available on top for reading mode.

  • The end-state if A and B both feel right — nothing here conflicts; B's metrics change lands first, A's surface work follows surface by surface.
  • Sequence suggestion: B now (one file) → A on Home as the pilot surface → roll the contextual header pattern to Agents/Tail → C last, once the dock's final height is known.

What you give up for good: an app-level brand moment and a guaranteed home for settings. If that starts to feel lost, the fallback identity slot is the Home contextual header itself (eyebrow becomes "SCOUT — FLEET").

+Maximum content, every element earning its pixels.
−Most moving parts; only worth it if A feels right on its own first.
